What is PNP Labs claiming?

PNP Labs presents a machine-checkable finite-certificate claim for P = NP over the locked NAND residual-slack package. The public theorem boundary is CheckPCCPackexp(GeneratePCCPack()) = accept ⇒ P = NP.

What does the browser check show?

It only shows that the bundled file matches the published SHA-256 digest. It does not validate the theorem claim, review the mathematical argument, or replace source regeneration.

Why does the site focus on the checker record?

The release claim is tied to executable artefacts. Reviewers need to know exactly which finite package, acceptance run, replay, final certificate, release gate, and report records are being relied on.

What is the locked NAND role?

The locked NAND construction encodes SAT into an exact-minimization threshold for a tagged multi-output direct-wire word. The report states that these instances have residual slack at most four, allowing the residual-band minimizer to be applied.

Where can a technical reviewer start?

For first-pass review, prioritise the claim boundary, locked NAND threshold, residual slack bound, residual-band exact minimization theorem, no-hidden-minimization scan, final package acceptance, and acceptance-replay linkage.

What is the difference between checking files and regenerating reports?

File checking compares the committed sealed bundle and checksum ledger. Regeneration checks that the recorded source/checker revision emits checker-accepted compact and full report records with the expected theorem statements and linkage.

What public description is accurate?

Recommended wording: "PNP Labs has released a machine-checkable finite-certificate claim for P = NP and is inviting independent technical verification." Avoid language implying external consensus unless that evidence exists.

Where do review requests and security disclosures go?

Technical review and source repository access requests go to review@pnplabs.com.au. Security issues and suspected artefact-integrity concerns go to security@pnplabs.com.au.
